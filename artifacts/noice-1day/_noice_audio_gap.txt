== Effort estimate ==
Premise: target environment is the OnePlus 6 phone running Westlake's own
ART11 inside dalvikvm64, with libohbridge.so as the JNI surface (per
project_dalvikvm64_bridge memory). Phone is arm64-v8a; the device's own
/system/lib64/libmedia_jni.so + libaudioclient.so are usable IF we can
dlopen them and provide the right context.

1. Audio focus stub  — 0.5-1 day
   Add three FindClass+RegisterNatives blocks for android.media.AudioManager
   (`native_requestAudioFocus`, `native_abandonAudioFocus`, `native_setMode`,
   `native_getStreamVolume`, `native_getStreamMaxVolume`). Make
   requestAudioFocus return AUDIOFOCUS_REQUEST_GRANTED unconditionally. This
   unblocks the FIRST gate noice currently fails — without it, no audio
   path ever runs.
   Risk: low. Pure no-op stub that just satisfies the focus contract.

2. NotificationManager stub for foreground service  — 1-2 days
   Stub `enqueueNotificationWithTag`, `cancelNotificationWithTag`,
   `createNotificationChannelsForPackage`, plus
   `ActivityManager.startForegroundService` shim. Required to prevent
   `Service.startForeground` ANR on Android 14+ targetSdk.
   Risk: low. Same model as existing ApkAssets / SystemProperties stubs.

3. AudioTrack JNI bridge (the load-bearing one) — 1-2 weeks
   Two paths:
   (a) Best-effort: dlopen the phone's /system/lib64/libmedia_jni.so and let
       its native_setup do its thing. On Pixel 7 Pro / OnePlus 6 this requires
       the process to have ACCESS_AUDIO permission AND a real Context binder
       — which we don't. Likely fails with E/AudioFlinger linker dlsym errors.
   (b) Reimplement the seven-method native_setup/native_write/native_play/
       native_stop/native_pause/native_release/native_get_min_buffer_size
       surface against AAudio (libaaudio.so on Android 8+, available on
       OnePlus 6). AAudio is a userland library, no SystemServer dependency,
       and gives us a stream we can write PCM into. This is the realistic
       1-2 week path. Stereo PCM 44.1/48kHz, S16/F32 — that covers ~all of
       Media3's DefaultAudioSink output formats.
   Risk: medium. Well-known shape of seven JNI methods; AAudio is documented.
   Output: noice plays sound (silence vs sound boundary).

4. MediaCodec JNI bridge (decoders) — 2-4 weeks for FOSS coverage
   Wide surface (~30 native methods). Two routes:
   (a) Wrap the phone's libmedia_jni.so MediaCodec entry points (same
       SystemServer dependency problem as AudioTrack option a).
   (b) Replace the codec backend with ExoPlayer's own NDK decoders (LibFLAC,
       LibOpus, LibVorbis, LibAv1) bundled as separate .so files in our
       installed app dir, and SUBSTITUTE noice's MediaCodecAudioRenderer with
       a FfmpegAudioRenderer at the Java level (not possible without
       repackaging the APK — out of scope for the read-only proof).
   (c) Pure stub that returns "no codec found" → noice's
       `MediaCodecRenderer.maybeInitCodec` throws
       `MediaCodecRenderer$DecoderInitializationException`. If we instead
       return success and accept-then-discard the decoded frames, we get
       silent playback which still progresses through the player state
       machine — useful as a smoke test.
   Effort to actually decode FLAC (the format noice uses for sound files):
   2-3 weeks if option (a) works on phone, 4+ weeks for option (b) if we
   need to write a custom AudioRenderer.
   Risk: high. This is the real bottleneck.

5. WakeLock / PowerManager  — already adequate (Binder stub); 0 days net
6. MediaSession / MediaSessionManager — 3-5 days for media-style notification
7. RingtoneManager (alarm path)  — 2-3 days; uses MediaPlayer under the hood,
   so blocked on item 4. AlarmRingerService can fall back to a hardcoded
   "ring.ogg" playback.

  - Add NotificationManager.enqueueNotificationWithTag stub (~20 lines)
  Result: noice's SoundPlaybackService can start as a foreground service and

  gate.

  - Stub MediaCodec to "always succeeds, discards frames" (option 4c) so
    that ExoPlayer's pipeline runs end-to-end with silent output
  Result: noice's UI fully usable, transport state changes, all sounds
  "play" but with no audible output (proves the gap is purely in the
  decoder, not in the rest of the audio chain).

Week 2:
  - Bridge MediaCodec to phone's libmedia_jni.so (option 4a) — try in the
    Westlake process first; if the SystemServer-dependent paths fail,
    decide between (b) and (c).
  - Add MediaSession + media-style notification.
  Result: actual sound from FLAC files. This is the proof-of-audio milestone.

Week 3+:
  - AlarmRingerService end-to-end (RingtoneManager → MediaPlayer JNI bridge,
    ~3 days on top of MediaCodec work).
  - Multi-stream mix (noice plays up to ~12 sounds simultaneously — Media3's
    DefaultAudioSink does this in Java, but we need to make sure AAudio
    handles 12 concurrent streams or we need a software mixer).
  - Audio effects (noice has crossfade + EQ — pure Java; will Just Work
    once AudioTrack does).

== Noice-specific quirks ==

1. Noice's own internal class is named
   Lcom/github/ashutoshgngwr/noice/engine/media/MediaPlayer; — same simple name
   as android.media.MediaPlayer. Anyone grep-ing for "MediaPlayer" will hit
   noice's wrapper. The framework MediaPlayer class IS referenced in the dex
   table, but as far as I can see noice's wrapper drives a Media3 ExoPlayer
   instance under the hood, not android.media.MediaPlayer. This is consistent
   with Media3 best practices for sound apps that need crossfade.

2. **ZERO bundled native libraries.** Most ExoPlayer apps ship libflac.so,
   libopus.so, libffmpeg_jni.so to handle non-platform formats. Noice does
   not — they rely entirely on the platform's MediaCodec for decoding. This
   means there is no "load the bundled .so and it Just Works" easy path;
   the audio decode side is 100% on the phone's libmedia_jni.so.

3. AlarmRingerService is independent of SoundPlaybackService and uses
   RingtoneManager (not Media3). For a 1-day proof we can ignore it — but
   if the user's actual workflow is "alarm goes off and noice rings", we'll
   need the MediaPlayer JNI surface working too.

4. Noice uses Media3 (`androidx.media3.*`), NOT the legacy
   `com.google.android.exoplayer2.*` namespace. APIs are similar but class
   names differ — any ExoPlayer-specific docs from before Mar-2022 won't
   match the dex strings.

5. The strings dump shows AudioFocusManager / DefaultAudioSink / AudioCapabilities
   are all in the dex as concrete classes. So at runtime, noice WILL execute
   these Java code paths — there's no Media3 lazy class-load that we can
   short-circuit by "just not creating the player".

6. `android.permission.FOREGROUND_SERVICE_MEDIA_PLAYBACK` is in the manifest,
   meaning targetSdk >= 34 enforcement applies. Without the foreground-service
   notification trick (item 2 in effort estimate), noice will be killed by
   the framework within 10s of start.

7. `android.hardware.audio.output` is a `uses-feature` requirement. Westlake's
   PackageInfoStub must report this hardware as present, or
   PackageManager.hasSystemFeature() will return false and noice's
   onboarding-flow will refuse to proceed.

== Summary of the gap ==
Closing the audio gap on day 1 is impossible — there is zero JNI bridging for
any android.media.* class today, and noice has 50+ android.media class refs
plus the entire androidx.media3 stack. A useful 1-day deliverable is the
two-stub patch (AudioFocus + NotificationManager) that lets noice's
foreground service successfully start and reach the AudioTrack-init gate.
Real audio output is a 1-2 week effort (AudioTrack via AAudio) plus a
2-4 week effort for MediaCodec decode coverage, with the 4-week worst case
if noice's FLAC path can't reuse the phone's libmedia_jni.so.
